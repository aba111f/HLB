Hi! This project is made as a final task for the subject "High Loaded Backend"

Our project provides exchange books between our users.

They need to register, log in. After users can add own book, or find exchange requests for getting another user's books.

In the porject we have CRUD for different models, and logs for views, already use JWT, and premission access on every view. 
And also we use kafka for asynchronous services, redis for cachings, elastic-search for searching system.
You can see documentation using swagger

The frontend our weak side. Some functions we didn't implement in frontend.

Ответить на вопросы: 
 

Как выбор структуры базы данных (SQL или NoSQL) влияет на дизайн CRUD API?
SQL более структурированный(табличный), NoSQL более гибкий. SQL использует жёсткие структурированные API запросы, часто нуждается в join, foreighn key, хорошо работает для станадртного CRUD. NoSQL отлично справляется с уже подготовленными запросами, позволяет быстро получать данные для конкретных сценариев.
  
Какие проблемы могут возникнуть при массовых обновлениях данных через API? 
Будет высокая нагрузка на сервер. Могут происходить Data hazards (WAW, WAR, RAW), response time будет бльшой, сервак может вообще упасть.


Почему важно использовать правильные HTTP-методы (GET, POST, PUT, DELETE), а не только POST? 
Это нужно для безопасности и валидации данных. Вторично: Так же на один и тот же URL мы можем посылать разные типы запросов, если использовать только POST, то для каждой функции нужен будет отдельный URL


Какие уязвимости могут возникнуть при хранении JWT на клиентской стороне? 
Если хранить JWT не в cookies, то злоумышленник может произвести атаку (Например XSS), и получить данные например local storage, в котором  может хранится JWT.

В каких случаях стоит ограничивать время жизни JWT, и какие проблемы это создаёт для UX? 
  Если не ограничивать время жизни JWT, то юзер будет постоянно использовать уже имеющиеся ключи. Это приводит к проблеме безоспасности, ведь злоумышлинники могут заметить какие одинаковые данные ты часто       отсылаешь(JWT), поймут, что это токен, и тогда смогут его скопировать.

Как логирование помогает в расследовании инцидентов безопасности? 
С помощью логов можно посмотреть структурированно куда совершаются атаки, какой тип запросов используется. Если мы видим, что злоумышлинник активно шлёт определённые запросы именно к определённому URL, то возможно в нём есть уязвимость.

В чём разница между горизонтальным и вертикальным масштабированием, и как это связано с кэшированием? 
Горизантальное маштабирование происходит за счёт колличества серверов, вертикальное, за счёт их мощности. Есть разыне методы кэширования, когда сервер один, то всё проще, ведь у тебя не может быть копиё данных, и тебе ен нужно распределять и запрашивать хеш с разных серверов. А так придётся разделять кеш, и когда надо совмещать пи запросе данных

Какой риск несут фоновые задачи при сбое очереди сообщений? 
Ну например если у мы захотим перевести деньги от одного пользователя другому, но тут очередь сообщений ломается, то может произойти так, что мы спишем деньги у одного клиента, но не отошлём другому, или наоборот.

Почему важно учитывать идемпотентность задач при их повторном выполнении? 
Потому что при какой либо ошибке, у нас может произойти повторения одного и того же действия, и это может привести к проблемам. Например, если сервер перегружен может произойти так, что он вполнить снятие срадств со счёта пользователя 2 раза вместо одного

Что сложнее поддерживать в большой системе: код или документацию? Почему? 
Сложнее поддерживать документацию, потому что если вы правильно настроили код, то всё работает, если нет, то будут ошибки и баги. Если вы не правильно сделали документацию, то система вам не выдаст ошибку, поэтому документацию приходиться перепроверять самому, или другому человеку, и то не факт, что вы не пропустите ошибку, которые оба не заметили

Какие плюсы и минусы у ручного написания README по сравнению с автогенерацией документации? 
Минусы в том, что ты сможешь точнее и лучше рассказать что у тебя за проект, и как с ним работать. Плюс один, но большой: автогенерация экономит твоё время.

Как документация помогает при онбординге новых разработчиков в команду? 
С помощью структурированной документации новый девелопер может точно понять, что у вас за проект, что уже есть, что можно или нужно добавить, и какие навыки нужны
